# ВАРИАНТЫ ФОРМАТОВ ДЛЯ СТАНДАРТА CHEFSMENU

## 1-й вариант   

[chefsmenu-menu-std.json](chefsmenu-menu-std.json)

Вариант, приближенный к текущему формату меню chefsmenu   
В нем все меню и товары с модификаторами вложены друг в друга.   

Это не удобно по многим причинам:
- мобификаторы могут повторяться и приходится их прописывать полностью много раз    
- поиск по вложенному списку модификаторов и товаров может быть сложным

Единственное преимущество:
- просто понять какой товар в какой группе, не визуализируя сам файл


## 2-й вариант

[chefsmenu-menu-std2.json](chefsmenu-menu-std2.json)

Более универсальный вариант, который может быть использован для любого типа меню.   
В этот формат мы будем преобразовывать все меню из разных источников (iiko, r-keeper и тд.)   

Преимущества:
- мобификаторы не повторяются
- группы модификаторов не повторяются
- поиск товаров, модификаторов и их групп гораздо проще реализовать
- файл в целом меньше
- его легко распарсить и визуализировать

Сложность:
- сложно визуально понять файл какие товары в какой группе, если не визуализировать файл


## РЕШЕНИЕ

Далее используем 2-й вариант как универсальный эталонный формат UNIMENU.    

### ПЕРВЫЙ ЭТАП РАЗВИТИЯ CHEFSMENU:

- Написать конвертер из UNIMENU в текущий формат chefsmenu.

### ВТОРОЙ ЭТАП РАЗВИТИЯ CHEFSMENU:

- Переделать структуру Базы Данных для хранения меню в формате UNIMENU
- Переписать парсер Внешнего меню так, чтобы сразу преобразовывать в UNIMENU.
